# #349 — AI 知识库文章生成器

**角色**：客户成功
**行业**：SaaS / 科技
**任务**：文档
**标识符**：`ai-knowledge-base-article-generator`

---

## 简介

知识库是永不休眠、永不疲惫、能同时服务每一位客户的24/7客户服务代表。对于拥有1000名客户和10人支持团队的SaaS公司，一个构建良好的知识库是支持基础设施中ROI最高的单一投资——如果以正确的内容、正确的阅读级别、用客户实际使用的语言构建，能够转移30-50%的入站工单。经济效益显而易见：一篇15美元成本的知识库文章每月转移5张工单，以每张22美元计算，每年节省1320美元。

然而，大多数SaaS公司的知识库是事后想法，是由支持客服在工单间隙抽空写出的内容构建而成，缺乏一致结构，用客户不会搜索的以产品为中心的术语编写，很少更新以反映产品变化或新发现的失败模式。知识库的理论潜力与其实际转移绩效之间的差距是巨大的——通常70-80%的潜在价值未被实现。

根本的生产瓶颈是写作时间。一篇高质量的知识库文章——结构良好、使用客户语言、涵盖主要失败模式、包含决策树或故障排除指南，并与搜索引擎索引逻辑正确集成——需要一位有经验的技术写作人员3-5小时才能完成。大多数SaaS公司没有专职技术写作人员，因此文章撰写负担落在已经满负荷的支持客服和CSM身上。

COCO通过充当知识库文章共同作者来消除这一瓶颈。CSM或支持客服提供原始材料——工单线程、解决方案笔记、产品描述或口头解释——COCO以适合团队知识库平台的适当格式，用面向客户的语言，以及所有使其可被发现、可快速浏览和可操作的结构性元素，生成高质量的文章草稿。

**COCO如何解决这一问题——分步说明：**

1. **输入处理**：COCO接受非结构化的原始输入——工单线程、Slack对话、通话笔记或简单的口头描述——并提取所记录的核心问题和答案。

2. **受众校准**：COCO调整技术深度和词汇以匹配目标受众（技术管理员vs业务用户vs终端用户），生成适合客户专业水平的版本。

3. **文章结构生成**：COCO应用知识库最佳实践来结构化文章：标题（搜索优化）、介绍、前提条件、分步说明（编号）、截图/视频占位符、故障排除部分和相关文章。

4. **SEO和可发现性优化**：COCO根据提供的工单数据中的语言，针对客户实际使用的搜索词优化标题和第一段。

5. **多格式输出**：COCO可以以Markdown（适用于Notion/Confluence）、HTML（适用于Zendesk）或纯文本格式生成文章，匹配团队的发布平台。

6. **质量评分**：COCO根据可读性标准评估自己的草稿，并标记任何过长、过于技术化或缺少关键结构元素的部分。

**可量化成果：**

- 从原始输入到可发布草稿的每篇文章时间：从3-5小时 → 20分钟
- 知识库文章质量评分（可读性、结构、完整性）：提升67%
- 有COCO生成文章的主题vs旧版文章的工单转移率：高出31个百分点
- 2人支持团队的月度文章产出能力：从4篇 → 28篇
- COCO生成文章的客户自助服务CSAT：4.4/5 vs 旧版文章2.8/5

**受益角色：**

- **支持客服**：将时间用于复杂问题解决，而非文档撰写——这是许多人在工单压力下感到耗尽精力并降级处理的任务。
- **客户成功经理**：将深厚的产品和客户知识转化为可扩展的自助服务资产，而无需成为技术写作人员。
- **支持负责人**：在不雇用额外写作人员的情况下实现显著更快的知识库增长、更好的文章质量和可量化的转移提升。
- **客户**：找到完整、准确、可读的常见问题答案，无需等待人工响应。

## 5个实用提示词

**提示词1 — 从工单线程生成文章**
```
从以下支持工单线程生成知识库文章。格式适配[Zendesk / Confluence / Notion——选一个]。

工单：[粘贴完整工单线程，包含客户问题和客服解决方案]

要求：
- 标题：使用客户使用的语言，针对搜索优化
- 受众：[技术管理员/业务用户/终端用户]
- 格式：介绍（涵盖内容）、前提条件、编号步骤、故障排除（前3个失败点）、相关文章（建议3个标题）
- 语调：直接、友好、无术语
- 包含：每个步骤的截图占位符和说明建议
```

**提示词2 — 从描述生成操作指南**
```
为以下任务撰写分步骤操作指南知识库文章。没有现有工单——我是主动撰写的。

任务：[描述任务——如"如何为我们的企业计划设置Okta SSO"]
产品：[产品名称]
受众：[具有中级技术知识的IT管理员]
常见错误：[列出3个常见错误]
相关功能：[列表]

包含：预计完成时间、前提条件清单、编号步骤、最后的"验证是否成功"步骤，以及链接到支持的"仍有问题？"部分。
```

**提示词3 — 故障排除指南**
```
为以下错误或问题撰写故障排除指南。该指南应帮助客户在80%以上的情况下诊断和解决问题而无需联系支持。

问题：[错误消息或问题描述]
常见原因：[列表]
识别具体原因的诊断步骤：[笔记]
每种原因的解决方案：[笔记]

以决策树故障排除指南的形式组织：
- 从最常见的原因及其解决方案开始
- 如果不起效，继续到下一个原因
- 全程使用清晰的"如果X，则Y"语言
- 结尾：如果这些都无法解决您的问题，请联系支持并提供[具体信息]以加快解决速度。
```

**提示词4 — 文章重写以提升清晰度**
```
为更好的清晰度、可浏览性和以客户为中心改写以下知识库文章。当前文章可读性评分低，因为：[问题——如"过多技术术语、大段文字、无编号步骤、全程被动语态"]。

当前文章：
[粘贴现有文章]

目标受众：[角色]
目标阅读级别：[年级水平 或 "非技术业务用户"]
将发布的平台：[平台]

应用知识库最佳实践。向我展示改善最大部分的前后对比。
```

**提示词5 — 批量文章规划**
```
根据以下未解决支持工单列表，识别本周撰写可产生最高转移影响的10篇文章。对每篇，提供：标题、估计月度转移量、文章类型，以及估计撰写复杂度（简单/中等/复杂）。

工单数据：[粘贴带标题的工单列表]
知识库中的当前文章：[现有文章标题列表]
平台：[Zendesk / Confluence / Notion]

输出为我可以带到每周团队会议的优先写作计划。
```
