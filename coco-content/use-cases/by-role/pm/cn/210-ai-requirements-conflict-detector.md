# 用例 #210：AI需求冲突检测器

**角色**：产品经理 | **行业**：企业软件、SaaS、金融科技、医疗科技 | **任务**：需求分析、利益相关方对齐、依赖关系管理

---
## 详细介绍

**痛点：冲突早就埋在那里了——没人发现，直到迭代第三周**

多利益相关方的产品开发，本质上是规模化的协调难题。一个产品经理在收集某个重要功能的需求时，可能收到来自六个不同团队的输入：销售团队需要支持企业大客户单点登录，安全团队要求基于角色的访问控制并附带审计日志，工程团队提出数据库性能约束，客户成功团队希望简化自助操作流程，法务团队要求明确的同意确认节点，而企业客户则要求批量导入且不能有数据校验延迟。这六个对话中，没有任何一方知道其他方说了什么——也没有人有时间翻阅全部六份笔记去寻找潜藏的地雷。

冲突在开发过程中才浮现，而非之前。工程师开始构建自助操作流程时，才发现它与基于角色的访问控制发生冲突。安全审查才发现批量导入绕过了同意确认节点。性能测试才发现审计日志要求打破了数据库性能约束。每一次这样的发现，至少需要1到2天的工程返工——但更常见的是触发需求文档的重写，重启规划流程，打乱迭代节奏，并需要新一轮利益相关方重新签字确认。

真正的代价不仅是返工时间。它还会侵蚀团队信任——一个功能在迭代中途反复改变形态，工程师对规划流程失去信心，利益相关方发现自己以为已达成共识的需求被悄悄删减时感到被忽视，而产品经理则把更多时间用于救火，而非真正领导产品方向。

**COCO如何解决**

COCO的AI需求冲突检测器能够同时分析来自多个来源的需求，在开发开始前识别矛盾点和依赖风险，并为利益相关方对齐生成解决方案框架。

1. **跨利益相关方需求解析**：接收来自多个来源的需求——会议记录、即时消息、邮件摘要、PRD评论、设计规格——并将其标准化为统一的需求模型。
   - 提取明确需求（"必须支持X"）和隐性约束（"不能影响Y"）
   - 为每项需求标注来源利益相关方和置信度
   - 识别用不同术语表述的、描述相同或相互冲突功能的需求

2. **冲突类型分类**：将检测到的冲突按类型分类，以便有针对性地解决。
   - **直接矛盾**：需求A声明X必须成立，需求B声明X不能成立
   - **资源冲突**：两项需求各自需要同一受限资源（带宽、数据库写入限制、API配额）
   - **优先级冲突**：两项需求都被标记为"必须有"，但在既定范围/时间线内无法同时交付
   - **依赖冲突**：需求A假设能力B已可用，但B尚未构建或正在变更中

3. **依赖关系图生成**：绘制需求之间的依赖关系，揭示隐藏的假设链条。
   - 展示哪些需求被其他需求阻塞
   - 识别循环依赖（A需要B，B需要A）
   - 标注高风险依赖枢纽需求——其变动会级联影响许多其他需求

4. **冲突严重程度评级**：按每个冲突若未解决将对交付造成的潜在影响进行评级。
   - 严重：将阻断功能交付或产生法律/技术上不可接受的结果
   - 高：若在开发过程中才发现，需要大量返工或范围变更
   - 中：造成实现层面的歧义，不同工程师可能以不同方式解决
   - 低：值得注意但不太可能引发问题的风格性或非阻断性不一致

5. **解决方案选项生成**：针对每个冲突，生成2到3个可能的解决路径及权衡分析。
   - 识别每个解决选项需要哪个利益相关方的需求做出改变
   - 估算每个解决路径对其他需求的下游影响
   - 标注需要向管理层上报的解决选项，区分于可由产品经理自主决定的情况

6. **对齐会议材料准备**：产出预结构化的冲突审查文档，可直接用于利益相关方会议。
   - 按严重程度和涉及的利益相关方组织冲突清单
   - 将每个冲突格式化为清晰的决策事项："我们需要在[选项A]和[选项B]之间做出决定。权衡如下。"
   - 生成决策记录模板，便于在会议中实时记录解决方案

**量化结果**

- **开发前冲突检测率**：开发启动前检测到70%到80%的冲突，而手动审查约为20%
- **迭代中途返工事件**：使用系统性冲突检测的团队减少约55%
- **利益相关方对齐会议效率**：冲突审查会议从90分钟缩短至40分钟（使用预结构化材料）
- **需求变更对迭代速度的影响**：从每季度约15%的速度损失降至5%以下
- **签字确认前的需求覆盖率**：跨利益相关方依赖关系记录覆盖率从约50%提升至90%以上
- **上线后需求回归率**：无检测冲突情况下上线的功能，上线后需求争议减少40%

**谁会受益**

- **产品经理**：在冲突变成昂贵的迭代中途发现之前就将其捕获——带着结构化解决方案议程进入利益相关方会议，而非临场应对
- **工程团队**：以内部一致、不自相矛盾的需求规格启动迭代——不再在第5天才发现两段需求相互矛盾
- **利益相关方（销售、法务、客户成功、安全）**：确保自己的需求被正确追踪，冲突被透明呈现——不再感到输入被忽视或悄悄删除
- **项目/项目群经理**：获得可用于交付规划和风险上报的依赖关系图和风险清单

## 实用提示词

**提示词1：全量多利益相关方需求冲突扫描**
```
我正在为 [公司名称] 的 [功能/产品名称] 构建需求规格说明。
我已从多个利益相关方收集了输入——请将所有内容一起分析，识别任何冲突、矛盾或依赖风险。

各利益相关方需求：

销售团队：
[粘贴销售需求/诉求]

工程团队：
[粘贴技术约束和工程需求]

安全/合规：
[粘贴安全和合规要求]

客户成功：
[粘贴客户成功需求和客户请求]

法务：
[粘贴法务要求]

设计：
[粘贴用户体验/设计约束]

请输出：
1. 所有检测到的冲突清单，按类型分类（直接矛盾/资源冲突/依赖冲突/优先级冲突）
2. 每个冲突的严重程度评级（严重/高/中/低）
3. 对每个"严重"和"高"级别冲突：提供2到3个解决选项及权衡分析
4. 展示需求间相互依赖关系的依赖关系图
5. 开发启动前必须做出决策的优先清单
```

**提示词2：两个团队需求对齐**
```
我有一个两个团队之间的需求冲突，需要在迭代规划前解决。

团队A的立场（[团队名称，例如：安全团队]）：
[完整粘贴他们的需求或约束条件]

团队B的立场（[团队名称，例如：工程团队]）：
[完整粘贴他们的需求或约束条件]

具体冲突：[用1到3句话描述你理解的冲突所在]

当前迭代时间线：[迭代启动时间，功能需要就绪的时间]

请帮我：
1. 客观清晰地重新表述冲突——双方真正的核心需求是什么，与他们提出的诉求有何区别？
2. 提供3种可能的解决方案，每种方案包含：
   - 团队A需要做出的改变
   - 团队B需要做出的改变
   - 引入的风险或权衡
   - 实现复杂度估算
3. 推荐最有可能被双方接受的方案，并说明理由
4. 识别哪些额外信息有助于更快解决这个冲突
5. 起草我在联合解决会议上向双方团队呈现时应使用的核心沟通要点
```

**提示词3：迭代启动前依赖风险评估**
```
我即将启动 [功能名称] 的迭代，想在开始前检查是否存在隐藏的依赖风险。

以下是我们当前的需求集合：[粘贴PRD或需求清单]

我已知的外部依赖：
- [依赖1]：[描述——例如："需要X团队的API，目前正在开发中"]
- [依赖2]：[描述]
- [依赖3]：[描述]

请分析并告诉我：
1. 我们规格说明中哪些需求依赖于上述外部事项——如果这些依赖延期或发生变化，每个需求会受到什么影响？
2. 我们自身需求内部是否存在我可能遗漏的依赖关系？
3. 需求中的"关键路径"是什么——哪项需求一旦延期会导致整个功能推迟？
4. 我们的哪些需求对其他系统或团队做出了未经明确验证的假设？
5. 针对前3个依赖风险：如果依赖未能按预期解决，缓解策略是什么？
```
