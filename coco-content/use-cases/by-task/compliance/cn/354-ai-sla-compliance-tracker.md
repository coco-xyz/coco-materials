# #354 — AI SLA 合规跟踪助手

**角色**: 客户成功
**行业**: SaaS / 科技
**任务**: 合规
**标识符**: `ai-sla-compliance-tracker`

---

## 简介

**痛点：SLA 违约在已经损害关系后才被发现**

服务级别协议是企业 SaaS 关系的合同支柱——它们定义了客户为之付费的绩效标准，也是供应商-客户关系中信任的基础。然而大多数 CS 团队被动地管理 SLA 合规：在电子表格中跟踪承诺，依赖工程团队对正常运行时间事件的告警，在违约发生数天后才在支持工单审查中发现响应时间违规。当 SLA 违规被正式识别时，客户已经经历了性能下降、形成了负面印象，而对于拥有专职采购和法律团队的企业客户来说，可能已经开始记录可支持合同信用申请或续约谈判筹码的过程。

跨多元化客户组合的 SLA 管理运营复杂性相当大。不同的客户层级通常承载不同的 SLA 承诺：年合同价值 20 万美元以上的企业客户可能有 99.9% 正常运行时间保证的合同，4 小时响应时间 SLA 和关键问题 24 小时解决 SLA；而年合同价值 3 万美元的中市场客户有 99.5% 正常运行时间承诺和次工作日响应要求。跨 100 个账户管理这些差异化承诺——根据每个客户的具体阈值跟踪正常运行时间事件、确保支持工单在每个层级的正确窗口内得到响应、识别正在接近合同信用阈值的客户——在标准工单系统和 CRM 工具无法很好解决的方式上具有运营复杂性。

主动沟通的失败尤其具有破坏性。当正常运行时间事件发生时，CS 团队通常通过客户投诉工单而非内部监控告警了解到——这意味着客户正在经历中断、决定联系支持、等待响应、并在 CS 团队甚至不知道事件正在发生之前就形成了负面印象。签署了 SLA 支持合同的企业客户有明确期望：供应商将在服务中断期间主动沟通、提供定期状态更新，并在事后提供事故报告。当这些沟通未能发生，或发生得太晚且质量差时，关系信任的破坏超过了技术 SLA 的违反。

**COCO 如何解决这一问题**

COCO 帮助 CS 团队主动管理 SLA 合规——监控承诺、在违约变成惯性之前生成告警、起草主动沟通，以及生成证明供应商责任的合规文档。

1. **SLA 承诺登记和跟踪**：COCO 帮助在客户组合中构建 SLA 承诺登记——记录每个客户的具体正常运行时间、响应时间和按层级划分的解决 SLA 承诺，并创建监控合规的跟踪框架。
   - 建立带有合同承诺文档的客户级 SLA 档案
   - 创建带有阈值告警的组合级 SLA 跟踪仪表板

2. **违约前风险标记**：COCO 分析当前绩效数据与 SLA 承诺，以识别在违约发生之前正在接近违约阈值的账户——实现主动干预。
   - 识别在月度正常运行时间阈值 20% 以内的账户
   - 标记表明 SLA 违约风险的支持队列响应时间模式

3. **事故期间主动客户沟通**：COCO 起草服务中断期间所需的实时客户沟通——初始事故通知、固定间隔的状态更新以及初步的事后沟通——以企业 SLA 义务所需的速度和质量。
   - 生成根据每个客户的层级和合同要求定制的事故通知模板
   - 在固定间隔生成带有当前状态和解决估算的更新消息

4. **事后报告生成**：COCO 起草企业 SLA 合同通常要求的正式事后报告——包括根本原因分析、事件时间线、恢复服务所采取的步骤以及实施的预防措施。
   - 以结构化、专业格式生成事后报告草稿
   - 根据客户层级和事故严重性定制深度和技术细节水平

5. **按客户层级的 SLA 合规报告**：COCO 按客户层级生成月度和季度 SLA 合规报告——显示正常运行时间绩效、响应时间合规率、解决时间合规率以及相对于 SLA 阈值的趋势。
   - 生成层级级别和账户级别的 SLA 合规摘要
   - 识别合规指标趋势预测未来违约方向的客户

6. **信用计算和主动提供**：COCO 帮助基于合同公式计算 SLA 信用义务、为受影响客户生成信用提供沟通，并为合规记录记录信用发放。
   - 基于中断持续时间和合同信用计划计算信用金额
   - 起草在客户索取之前保护关系的主动信用提供沟通

**可量化的成果**

- **违约发现时间**：使用 COCO 辅助 SLA 监控的 CS 团队比依赖被动事故报告的团队平均提前 14 天识别即将到来的违约
- **事故期间客户沟通速度**：COCO 辅助的事故沟通起草将首次客户通知时间从 45-90 分钟减少到 8-12 分钟
- **事后报告完成率**：拥有结构化事后报告流程的组织为 94% 的符合条件事故完成报告，vs. 没有结构化流程的团队的 35% 完成率
- **信用索赔纠纷**：在客户索赔之前主动提供信用，与等待客户发起信用请求的组织相比，将信用纠纷事件减少 78%
- **SLA 驱动的流失减少**：在事故期间收到主动 SLA 管理沟通的账户，续约时的流失率比事故期间没有收到主动沟通的账户低 22%

**受益角色**

- **管理有正式 SLA 承诺的企业账户的客户成功经理**：需要结构化工具来监控合规、起草沟通并记录跨具有不同合同要求的多个账户的补救措施
- **CS 运营团队**：负责设计和维护 SLA 合规基础设施——跟踪系统、报告流程和沟通模板——需要结构化框架大规模运营化 SLA 管理
- **工程和技术支持团队**：管理 SLA 合规的正常运行时间和事故响应方面，需要满足合同标准的高质量客户沟通模板，而无需为每次事故进行定制写作
- **客户成功副总裁和总监**：将 SLA 合规作为客户健康和合同义务的组成部分负责——需要组合级别的 SLA 风险集中在哪里的可见性

## 实用提示词

**提示词 1 — SLA 承诺登记和跟踪框架**
```
我需要为我的 [X] 个企业账户建立结构化的 SLA 承诺跟踪系统。目前，SLA 承诺分散在合同文件中，我没有对不同账户承诺内容的集中视图。请帮我设计这个系统。

我们的 SLA 层级和典型承诺：
- 企业层级（年合同价值 10 万美元以上）：[描述 SLA——例如 99.9% 月度正常运行时间，P1 响应 <4 小时，P2 响应 <24 小时，5 个工作日内根本原因报告]
- 中市场层级（年合同价值 3-10 万美元）：[描述 SLA]
- 标准层级（年合同价值 <3 万美元）：[描述 SLA]

请帮我设计：
1. 每个客户账户的 SLA 档案模板（每个账户记录什么）
2. 月度 SLA 合规跟踪框架（如何衡量和记录合规情况）
3. 告警阈值：在什么绩效水平上我应该将账户标记为"接近违约"？
4. 我可以为我的 CS 总监生成的月度 SLA 合规报告模板
5. 处理 SLA 例外的流程——当我们提供信用或补救时需要什么文档？
```

**提示词 2 — 主动事故客户沟通**
```
我们目前正在经历影响 [数量或层级] 客户的服务事故。我需要立即起草初始客户通知，并在事故持续期间建立沟通节奏。

事故详情：
- 受影响内容：[描述受影响的服务或功能]
- 受影响客户：[哪些层级/账户，以及具体如何受影响]
- 当前状态：[关于原因和预计解决时间我们目前知道的]
- 初始发现时间：[何时检测到事故]
- 预计解决时间：[如果已知——如果未知请诚实]
- SLA 影响：[有客户正在接近或违反正常运行时间 SLA 吗？]

请起草：
1. 初始事故通知邮件（在事故检测后 15 分钟内发送）
2. 事故持续期间每 [30/60/90] 分钟发送的状态更新模板
3. 服务恢复时发送的解决通知邮件
4. 为有正式事后报告要求的企业账户发送的初步"我们将跟进完整报告"消息
5. 事故持续期间的内部 CS 团队状态更新

所有客户沟通的语气：主动、诚实、具体且冷静。不使用公司套话。
```

**提示词 3 — 事后报告**
```
我们在 [日期] 经历了一次 [描述事故——例如"导致 2 小时服务中断的数据库故障切换"]。我需要为有合同事后报告要求的企业客户编写正式事后报告。

事故时间线：
- 事故开始：[时间]
- 检测时间：[时间——以及如何被检测到]
- 解决时间：[时间]
- 总持续时间：[X 小时，X 分钟]
- 影响：[哪些服务受影响，哪些客户受影响，可量化的影响]

根本原因：[描述导致事故的原因]
促成因素：[延长事故持续时间或严重性的任何因素]
解决步骤：[为恢复服务所做的工作]
预防措施：[为防止复发已做出或将做出的变更]
SLA 状态：[这是否违反了任何客户的 SLA？哪些客户应获得信用？]

请起草正式事后报告，要求：
1. 以事故和影响的直接摘要开场（不废话）
2. 以时间顺序格式提供时间线
3. 以足够详细的技术内容解释根本原因以展示严谨性，但对非技术客户联系人来说足够易懂
4. 具体描述预防措施——不使用"我们将改善监控"这样的通用承诺
5. 如适用，承认 SLA 影响和信用义务
6. 以负责任且透明而非防御性的语气撰写
```

**提示词 4 — 月度 SLA 合规报告**
```
我需要为我的 CS 总监生成 [月份，年份] 的月度 SLA 合规报告，以及与请求月度合规摘要的企业账户分享。

正常运行时间数据：
- 整体平台正常运行时间：[X.XX%]（SLA 承诺：[X.XX%]）
- 本月事故：[X] 次事故，总停机时间：[X 小时，X 分钟]
- 按账户层级：[企业：X.XX%，中市场：X.XX%，标准：X.XX%]
- 违反其特定 SLA 的任何账户：[如适用列出]

支持响应时间数据：
- P1（关键）：[X%] 工单在 SLA 内响应（[目标小时]）——[X] 张工单总计
- P2（高）：[X%] 在 SLA 内（[目标小时]）——[X] 张工单总计
- P3（中）：[X%] 在 SLA 内（[目标小时]）——[X] 张工单总计
- 显著超出 SLA 的任何异常工单：[描述]

本月发放的信用：[金额、客户、原因]

请生成：
1. 本月 SLA 合规的高管摘要（绿色/黄色/红色状态及理由）
2. 账户级别标记：哪些特定账户本月 SLA 体验最差？
3. 趋势分析：本月与过去 3 个月相比如何？
4. 前瞻性风险标记：哪些账户或领域显示出预示未来 SLA 压力的模式？
5. 适合与请求月度摘要的企业账户分享的面向客户版本报告
```

**提示词 5 — SLA 信用提供沟通**
```
我们的平台在 [日期] 经历了持续 [X 小时，X 分钟] 的中断。根据我们与 [客户名称] 的合同 SLA，此违规触发了服务信用义务。

合同详情：
- 客户：[名称、层级、年合同价值]
- SLA 承诺：[X.XX%] 月度正常运行时间
- 本月实际正常运行时间：[X.XX%]
- 合同信用公式：[例如，超出 SLA 的每个百分点停机时间按月费的 10%，最高 30%]
- 应付信用金额：[$X]
- 信用应用方式：[在下一张发票上抵扣/单独付款/其他]

关系背景：
- 账户健康：[绿色/黄色/红色]
- 续约日期：[X 个月后]
- 此客户在中断期间是否直接受到影响？[是/否——描述影响]
- 他们在事故期间的反应：[冷静/沮丧/升级]
- 他们是否就中断联系过我们？[是/否]

请起草主动信用提供沟通，要求：
1. 以对中断及其对其运营的影响承担全责开场
2. 在他们不得不询问之前，清晰说明他们应获得的信用及其应用方式
3. 参考已沟通的任何事后报告或预防措施
4. 不显得事务性——这是一次关系恢复沟通，不是计费通知
5. 以对其持续成功的承诺以及如果他们想进一步讨论事故的邀请结尾

语气：温暖、负责任、直接。长度：200-250 字。
```
