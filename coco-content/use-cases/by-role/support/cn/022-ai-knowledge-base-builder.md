# Use Case #022: AI Knowledge Base Builder

**Role**: Support Lead / Technical Writer / Knowledge Manager | **Industry**: SaaS, Technology, E-commerce, Any Company with Customer Support | **Task**: Knowledge Base Creation, Article Writing, Content Gap Analysis, Self-Service Optimization

---

## 社媒文案

我们有一个340篇文章的知识库。听起来不错，直到你看数据：127篇文章超过18个月没更新。43篇文章引用了已经不存在的功能。搜索功能对38%的客户查询返回"无结果"。而我们的坐席一遍又一遍地回答同样的50个问题，因为这些问题的KB文章要么缺失、要么过时、要么写得太差客户看不懂。

结果？只有12%的工单被自助服务拦截。成熟KB的行业基准：40-60%。

建设知识库内容是每个客服团队都知道应该做却永远没时间做的事。每篇高质量文章需要2-3小时来撰写、审核、截图和发布。积压了200+篇待写文章，一个技术写作者需要整整一年才能追上。

COCO的AI Knowledge Base Builder清除了积压：
- 分析6个月的已解决工单，识别客户最常问的前200个问题
- 与现有KB交叉对比，找到缺口和过时内容
- 从已解决工单的解决方案生成文章草稿，格式化为分步指引、截图占位符和故障排除树
- 技术写作者的工作从写作变成审核和发布

KB文章：从340篇（127篇过时）到520篇（全部当前）。自助服务拦截率：从12%升至41%。工单量：下降29%。最好的客服交互是那些根本不需要发生的。

## 详细介绍

**痛点：根本帮不了任何人的知识库**

大多数公司有知识库。但很少有高效的。"有一个KB"和"有一个真正能拦截工单的KB"之间的差距是巨大的，衡量标准是内容质量、覆盖率、新鲜度和可搜索性。

内容债务问题是普遍的。产品演进的速度快于文档团队的更新速度。功能改名、工作流变更、新集成上线——描述旧行为的KB文章变成了积极有害的，将客户引向错误的路径，产生的工单比它们阻止的还多。

KB维护的经济账很有挑战性。一篇全面、清晰、富含截图的帮助文章需要经验丰富的技术写作者2-3小时。维护它（检查准确性、更新截图、添加新步骤）每年每篇文章还需要1-2小时。500篇文章意味着每年仅维护就需要500-1000小时——还没算写新内容。

**COCO如何解决**

COCO的AI Knowledge Base Builder将已解决的客服工单转化为已发布的知识库内容，保持KB全面且当前。

1. **工单到文章挖掘**：COCO分析已解决工单历史：
   - 识别最常被问到的问题（按量和搜索查询）
   - 提取坐席为每个问题提供的解决方案
   - 将相似工单分组以找到标准解决方案
   - 识别没有现有KB文章的问题

2. **文章生成**：对每个识别的缺口，COCO生成：
   - 清晰、无术语、搜索优化的标题
   - 带编号步骤的分步指引
   - 带有截图描述的截图占位符
   - 复杂问题的故障排除决策树
   - 相关文章和交叉引用
   - 简单问题的FAQ格式

3. **内容新鲜度管理**：COCO持续监控内容漂移：
   - 将KB文章与当前产品行为对比
   - 标记引用已弃用功能、旧UI或变更工作流的文章
   - 生成包含当前信息的更新草稿
   - 跟踪文章年龄并安排审查

4. **搜索优化**：COCO改善KB的可发现性：
   - 为文章添加同义词和替代表述（使"登录不了"匹配"无法登录"）
   - 生成为内部搜索优化的元描述
   - 基于搜索分析建议文章重组
   - 识别无结果的"死胡同"搜索并为其创建内容

5. **多格式内容**：除文字文章外，COCO还创建：
   - 交互式故障排除指南（如果X，尝试Y；如果不行，尝试Z）
   - 新功能快速入门指南
   - 屏幕录制的视频脚本大纲
   - 聊天机器人就绪的问答对
   - 应用内提示框内容

6. **效果分析**：COCO跟踪KB表现：
   - 文章浏览量、搜索命中率和自助解决率
   - 高浏览量但低满意度的文章（需要重写）
   - 无结果的搜索词（内容缺口）
   - 工单到文章的关联（哪些文章真正阻止了工单）

**可量化的结果**

- **KB文章覆盖率**：从340篇（127篇过时）到520篇（全部当前）
- **自助拦截率**：从12%升至41%
- **工单量**：减少29%
- **文章创作时间**：从2-3小时降至30分钟（仅审核和发布）
- **KB新鲜度**：100%文章在90天周期内审查
- **搜索"无结果"率**：从38%降至8%
- **客户对自助服务的满意度**：从2.8/5升至4.1/5

**受益角色**

- **客户**：自己找到答案，更快，全天候
- **客服坐席**：更少重复工单，更多时间处理复杂问题
- **技术写作者**：从写作者转变为编辑，更高影响力的工作
- **客服领导**：更低的每次解决成本，更好的CSAT，可扩展的客服

## 实用提示词

**提示词 1: 从已解决工单生成KB文章**
```
基于这些关于同一问题的已解决工单创建一篇面向客户的知识库文章。

已解决工单：
工单1：客户问："[问题]" - 坐席解决方法："[解决步骤]"
工单2：客户问："[问题]" - 坐席解决方法："[解决步骤]"
工单3：客户问："[问题]" - 坐席解决方法："[解决步骤]"

撰写KB文章包含：
1. 清晰、搜索友好的标题
2. 简要描述用户何时会遇到此问题
3. 分步解决方案（编号、清晰、假设没有技术背景）
4. [截图：描述需要截取的内容] 在视觉指导有帮助的地方放置占位符
5. 故障排除部分："如果以上步骤无效，请尝试..."
6. 相关文章部分（建议2-3个相关主题）

语气：友好、有帮助。在初中阅读水平撰写。避免术语。
产品名称：[你的产品名]
```

**提示词 2: KB内容缺口分析**
```
分析这些客服工单类别并识别知识库内容缺口。

过去90天按量排名前20的工单类别：
1. [类别] - [X张工单] - KB文章是否存在：[是/否]
2. [类别] - [X张工单] - KB文章是否存在：[是/否]
[...继续列出全部20个]

搜索"无结果"的前20个KB搜索查询：
1. "[搜索查询]" - [X次搜索]
2. "[搜索查询]" - [X次搜索]
[...继续]

生成：
1. 需要创建的文章优先级列表（最高工单拦截潜力优先）
2. 每篇需要的文章：建议标题、大纲和预估复杂度（简单FAQ / 分步指南 / 故障排除指南）
3. 需要更新的现有文章（基于高量+已有文章）
4. 建议的文章合并（应该合并的多篇文章）
5. 如果填补这些缺口的预估工单拦截量（基于量数据）
```

**提示词 3: 重写表现不佳的KB文章**
```
这篇KB文章有高流量但低满意度评分。重写使其更清晰、更有帮助。

当前文章：
[粘贴当前文章文本]

表现数据：
- 月浏览量：[X]
- 有帮助评分：[X]/5
- 常见反馈：[如有，粘贴客户反馈]
- 查看后仍提交工单的浏览者比例：[X]%

重写文章包含：
1. 更清晰的标题（用客户的话说，这解决什么问题？）
2. "开始之前"部分（前置条件、需要准备什么）
3. 简化的分步指引（更短的句子，每步一个操作）
4. 步骤复杂处的视觉指导占位符
5. "常见问题"故障排除部分
6. 明确的成功标准（"当...时，你就知道成功了"）
7. "仍需帮助？"部分，附联系方式

对比原文和重写，标注具体改进。
```
