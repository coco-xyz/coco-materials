# Use Case #208: AI PRD Generator

**Role**: Product Manager | **Industry**: SaaS, Enterprise Software, Consumer Tech, Fintech | **Task**: Product Requirements Documentation, Stakeholder Alignment, Feature Scoping

---
## 详细介绍

**痛点：写PRD是那件不断抢占本职工作时间的事**

产品经理最不可替代的价值是判断力——决定做什么、为什么做、按什么顺序做。然而大多数产品经理每写一份PRD就要花6到10小时，消耗掉的正是迭代前本该用来验证决策、对齐利益相关方和思考策略的时间窗口。对于两周迭代的团队来说，这意味着每个迭代周期有三分之一被文档开销吞噬，而非真正的产品思考。

质量问题叠加时间问题，雪上加霜。在时间压力下写出的PRD往往不完整——边界情况缺失、验收标准模糊，或者与驱动这个功能的用研和战略背景脱节。工程师提出澄清问题导致迭代启动延期，设计师按模糊需求出图，QA按与原意不符的验收标准测试。PRD中每一处模糊，下游都要用30到60分钟的会议来解决——而这些会议往往发生在迭代中期，改变方向的代价已经很高。

团队通常用更短、更精简的文档来应对——但以速度换来的"精简"PRD，最终通过Slack讨论、同步说明会议和迭代中途的范围变更，制造出同等的协作开销。清晰、完整、共识一致的需求文档没有捷径可走——但到达那里有更快的路。

**COCO如何解决**

COCO的AI PRD生成器能够将原始输入——会议记录、研究发现、战略目标、竞品参考——在极短时间内转化为结构完整的PRD草稿。

1. **上下文到结构的转换**：将非结构化输入（粗糙的笔记、语音备忘录、要点列表）自动整理为完整的PRD结构。
   - 自动填充标准PRD章节：背景、问题陈述、目标、用户故事、需求、验收标准、范围外事项、待定问题
   - 从上下文中提取产品经理未明确陈述的隐性需求
   - 标记输入不足、无法生成可靠规格说明的区域

2. **用户故事生成**：从功能描述中生成规范的用户故事，包含用户画像、操作和结果。
   - 从同一功能描述中生成主要用户故事和相关边界情况故事
   - 采用标准格式："作为[用户]，我希望[操作]，以便[收益]"
   - 将每个故事与驱动它的用户痛点或研究证据关联

3. **验收标准编写**：为每项需求生成具体、可测试的验收标准。
   - 区分功能性验收标准、边界情况和非功能性需求（性能、无障碍、安全）
   - 编写格式可直接被QA和工程师使用，无需二次解读
   - 复杂交互需求采用"给定/当/则"格式

4. **假设与风险标注**：识别需求中内嵌的假设及这些假设被证伪时的相关风险。
   - 挖掘对其他团队、服务或功能的依赖关系
   - 标记依赖未验证假设的需求
   - 生成带可能性和影响程度评级的风险清单章节

5. **范围边界定义**：明确定义哪些在范围内、哪些不在——防止开发过程中的范围蔓延和对齐失败。
   - 基于既定目标，识别应被明确排除在外的可能范围外事项
   - 提示工程师或利益相关方可能提出的相关功能，并建议是否纳入
   - 创建"未来考虑事项"章节，用于记录延期范围

6. **可提交格式输出**：输出格式化完整的PRD，可直接粘贴到Confluence、Notion、Google Docs或任何文档系统，无需重新排版。
   - 生成统一的标题层级、需求矩阵表格格式和章节编号结构
   - 包含元数据块（作者、日期、版本、状态）和评审流程占位符

**可量化的结果**

- **PRD起草时间**：从6到10小时 → 60到90分钟（效率提升80%至85%）
- **工程师提出的澄清问题**：因验收标准完整度提升，减少约50%
- **PRD覆盖率评分**（已定义章节数、每个故事的验收标准数量）：比手写PRD提升40%
- **因需求不完整导致的迭代启动延期**：从影响约30%的迭代降至10%以下
- **利益相关方评审轮次**：正式签字前平均减少2.1轮修改
- **产品经理释放的时间**：每个迭代周期节省4到6小时，可用于研究、战略和用户沟通

**受益角色**

- **产品经理**：文档工作减少80%，将更多精力投入真正重要的判断、研究和利益相关方沟通
- **工程团队**：以完整、无歧义的需求启动迭代，大幅减少迭代中期的范围澄清和返工
- **QA/测试工程师**：直接从PRD获取可测试的验收标准，减少需求解读开销
- **产品设计师**：有清晰的需求基础可供设计，减少在范围和边界情况上的来回确认

## 实用提示词

**提示词1：基于会议记录和研究结果生成完整PRD**
```
我需要为 [公司名称] 的一个新功能撰写PRD。以下是我的原始输入：

功能概念：[用2到5句话描述该功能]

战略背景：[我们为什么要做这件事——它支持哪个目标或OKR]

用户研究洞察：[粘贴相关研究发现、用户原话或痛点]

利益相关方需求：[销售、客户成功、工程或管理层的关键输入]

竞品参考：[被参考的竞品功能（如有）]

约束条件：[技术约束、时间线、资源限制]

请生成包含以下章节的完整PRD：
1. 背景与问题陈述
2. 目标与成功指标
3. 用户故事（主要场景及边界情况）
4. 功能性需求及验收标准
5. 非功能性需求（性能、安全、无障碍，按相关性填写）
6. 范围外事项
7. 假设与风险
8. 待定问题
9. 未来考虑事项

请格式化为可直接粘贴至Confluence/Notion的输出格式。
```

**提示词2：验收标准深度补全**
```
以下是 [功能名称] 的用户故事，需要为其编写完整、可测试的验收标准。
请为每个故事编写"给定/当/则"格式的验收标准，包含：
- 主要成功路径
- 关键边界情况（空状态、错误状态、边界条件）
- 非功能性考量（加载状态、错误提示信息、无障碍性）

用户故事：
1. [故事1]
2. [故事2]
3. [故事3]

技术背景：[描述相关技术约束或后端行为]
用户群体：[使用该功能的用户类型]
平台：[Web / 移动端 / 两者均有]

另外，请标注哪些故事的验收标准暴露出我在迭代规划前需要解决的歧义点。
```

**提示词3：PRD差距分析与改进**
```
我已完成 [功能名称] 的PRD草稿，想在分享给工程团队前对其进行压力测试。
请审查以下草稿，识别：

1. 缺失的验收标准——没有明确可测试标准的需求
2. 未定义的边界情况——需求未覆盖的交互场景或状态
3. 范围歧义——工程师可能对需求产生不同解读的区域
4. 缺失的依赖关系——该功能运行所需但未列出的前提条件
5. 假设风险——依赖未经验证的用户行为或系统能力假设的需求
6. 需求冲突——相互矛盾的需求条目

针对每个发现的问题，请给出具体的修改建议。

[在下方粘贴PRD草稿]

补充背景：工程团队规模为 [X] 人，迭代周期为 [Y] 天，计划在 [日期] 启动本迭代。
```
